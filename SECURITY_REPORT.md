# Security Assessment Report
**Project:** Apollo University Portal  
**Date:** 03 February 2026  
**Status:** Critical Issues Identified

---

## 1. Executive Summary
The application is currently in a prototype/development state using local JSON files for data persistence. While functional for a demo, it contains **critical security vulnerabilities** that make it unsafe for any production or public-facing deployment. The most severe issue is the storage of user passwords in plain text.

---

## 2. Critical Vulnerabilities (Must Fix)

### 🔴 1. Plain Text Password Storage
**Location:** `server/data/users.json`, `server/routes/auth.js`  
**Issue:** Passwords are stored and compared as plain text strings (e.g., `"Test@123"`).  
**Risk:** If the server is compromised or the `users.json` file is leaked, **all user accounts are immediately compromised**. There is no hashing or salting protection.  
**Remediation:** Implement `bcryptjs` to hash passwords before saving them. Only compare hashes during login.

### 🔴 2. Weak & Hardcoded JWT Secret
**Location:** `server/routes/auth.js`  
**Issue:** The signing key for JSON Web Tokens defaults to `'secret'` if an environment variable is not present:
```javascript
jwt.sign({ id }, process.env.JWT_SECRET || 'secret', ...)
```
**Risk:** Attackers can easily guess this default secret (`'secret'`) and forge their own authentication tokens, allowing them to impersonate **any user (including Admins)**.
**Remediation:** Remove the fallback. Enforce the presence of a strong, random `JWT_SECRET` in the `.env` file.

### 🔴 3. Insecure Local File Storage
**Location:** `server/data/*.json`
**Issue:** The application uses simple JSON files as a database.
**Risk:**
- **Data Corruption:** Concurrent writes (two users saving at once) will overwrite each other's data.
- **Exposure:** JSON files in the source tree might be accidentally committed to version control (GitHub/GitLab), exposing all user data.
**Remediation:** Migrate to a proper database system like **MongoDB** or **PostgreSQL**.

---

## 3. High Risk Issues

### 🟠 1. Unrestricted CORS Policy
**Location:** `server/index.js`
**Issue:** `app.use(cors())` allows requests from **any origin**.
**Risk:** Malicious websites can make authorized requests on behalf of your users if they visit such a site while logged in (CSRF-like scenarios).
**Remediation:** Restrict CORS to specific trusted domains (e.g., `http://localhost:5173` or your production domain).

### 🟠 2. Missing Security Headers
**Location:** `server/index.js`
**Issue:** Standard security headers (Helmet) are missing.
**Risk:** Leaves the app vulnerable to common web attacks like XSS (Cross-Site Scripting) and clickjacking.
**Remediation:** Install and use the `helmet` middleware.

### 🟠 3. Lack of Rate Limiting
**Location:** Global / Auth Routes
**Issue:** There is no limit on how many requests a user can send.
**Risk:** An attacker can brute-force passwords by sending thousands of login attempts per second without being blocked.
**Remediation:** Implement `express-rate-limit` on sensitive endpoints like `/api/auth/login`.

---

## 4. Other Observations

- **Input Validation:** Minimal input validation is performed on the backend. Malformed data could crash the server or corrupt the JSON files.
- **Error Handling:** Some errors log full error objects to the console, which might inadvertently leak sensitive path or data info in production logs.

---

## 5. Recommended Action Plan

1.  **Immediate (Phase 1):**
    *   Install `bcryptjs`.
    *   Write a migration script to hash all existing cleartext passwords in `users.json`.
    *   Update `auth.js` to hash passwords on signup/creation and compare hashes on login.
    *   Set a strong `JWT_SECRET` in `.env`.

2.  **Short Term (Phase 2):**
    *   Configure `cors` options.
    *   Add `helmet` and `express-rate-limit`.

3.  **Long Term (Phase 3):**
    *   Migrate from JSON files to **MongoDB Atlas** (Cloud).
    *   Implement strict Zod/Joi validation for all API inputs.
